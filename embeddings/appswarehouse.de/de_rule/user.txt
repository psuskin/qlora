Dies ist die Beschreibung des Moduls "user" mit dem Namen "Benutzer": Diese App dient der Pflege der Benutzer, die Ihre ClassiX®-Applikationen benutzen dürfen. Neben den Anmeldeinformationen eines Benutzers, wie ID und Passwort, können hier auch die Zugriffsrechte definiert werden.Im Regelfall sind es die "eigenen" Mitarbeiter der im System definierten Mandanten, die Benutzer sind. Ein Benutzer zu sein bedeutet also, dass eine Person - neben seiner Rolle als Mitarbeiter - die zusätzliche Rolle als Benutzer hat. Die Erfassung eines Benutzers ist also im eigentlichen Sinne die Erfassung einer Benutzerrolle.Damit ist es auch möglich, Ansprechpartnern fremder Firmen Benutzerrollen zuzuweisen. Ansprechpartner fremder Firmen haben als Person - im Gegensatz zu einem Mitarbeiter - keine Mitarbeiterrolle als Angestellter eines "System Mandanten", sondern eine Ansprechpartnerrolle als Angestellter einer fremden Firma. Somit kann man auch diesen Personen eine weitere Rolle als Benutzer ihres Systems zur Verfügung stellen. (z.B. ihren Steuerberatern oder Wirtschaftsprüfern - aber auch vorübergehenden Leiharbeitern - die weiterhin Ansprechpartner der Fremdfirma sind und bleiben).Sollten solche "fremden" Benutzer auch Tätigkeiten ausüben, bei denen z.B. Belege erstellt werden mit An-/Ausgabe von Kontaktdaten, ist es sinnvoll, diese Kontakdaten (E-Mail, Telefon Durchwahlnummer) - rein in der Rolle als Benutzer - direkt an der Benutzerrolle zu hinterlegen. Ansonsten besteht die Gefahr, dass das System beim Ausdruck eines Dokuments oder Versenden einer E-Mail die Kontaktdaten der "fremden" Ansprechpartnerrolle übernimmt.Auch im Falle, dass z.B. die Mitarbeiter im Vertrieb nur über eine Sammel-E-Mail (z.B. sales@myCompany.com) kontaktiert werden sollen, kann man diese Sammel-E-Mail für jeden dieser Benutzer an seiner Benutzerrolle hinterlegen. In der Mitarbeiterrolle selber können dann die eigentlichen Kontaktdaten für den internen Firmenverkehr unverändert und unverwechselbar bestehen bleiben.Neben den Benutzerrolle spezifischen Kontaktdaten, kann einem Benutzer auch eine eigene erste Tätigkeitsstätte zugeordnet werden.Neben Benutzerrollen, die einer natürlichen Person zugewiesen werden können, braucht man gelegentlich auch Benutzerrollen, die nicht einer bestimmten Person dauerhaft zugewiesen werden können. Das ist zum Beispiel der Fall, wenn das System seitens "Dritter" benutzt werden soll, z.B. bei anonymer Nutzung über eine Schnittstelle (API). Für solche Fälle können Benutzerrollen auch Firmen (juristischen Personen) zugewiesen werden. Ob die Zuordnung zu der eigenen Firma (z.B. für Nachtläufe) oder zu einer fremden Firma vorgenommen wird (z.B der Betreiber des online shops, der Daten übermittelt) ist hierbei nicht relevant.Grundsätzlich ist es auch möglich, Benutzerrollen ohne irgendeine Zuordnung zu einer natürlichen oder juristischen Person anzulegen. Die Änderungshistorie von Daten weist dann aber auch Lücken auf. Dies ist die Beschreibung der Funktionalität des Moduls "user" mit dem Namen "Benutzer" bezüglich Eingabefenster: Dieses Fenster dient der Pflege von Benutzern. Sie können einen Benutzer auswählen, in dem Sie im Feld Benutzer-ID das gesuchte Kürzel eingeben und mit RETURN bestätigen. Wenn Sie das Listenfenster aufrufen wollen, dann können Sie den Menüpunkt "Suchen" auswählen. Der Login Name (Benutzer-ID) und das Passwort muss eingegeben werden. Weiterhin kann die Personalnummer und auch eine zeitliche Beschränkung für die Gültigkeit des Logins hinzugefügt werden. Im unterem Bereich des Fensters besteht ferner die Möglichkeit, eingeschränkte Zugriffsrechte zu definieren. Sollten Sie sich beim Ersteinsatz des Systems Ihr Login hier gerade neu angelegt haben, dann melden sie sich bitte noch einmal neu im System an. Weil dem Login möglichst eine reale Person zugeordnet werden sollte, ist es empfehlenswert, zunächst die Personendaten in der ClassiX®-Partnerverwaltung anzulegen. Dieses können Sie aber auch später nachholen. Sie sollten jedoch generell bedenken, dass ´Benutzer´ nur die Rolle einer Person ist und eine Person durchaus mehrere Rollen haben kann. Wenn Sie erst die Person anlegen und dieser dann anschließend eine oder mehrere Rollen zuweisen, können Sie einerseits doppelte Eingaben vermeiden und haben andererseits Ihre Daten stets gut strukturiert. Systembenutzer müssen jedoch nicht gleichzeitig als Person angelegt sein! Dies ist die Beschreibung der Funktionalität des Moduls "user" mit dem Namen "Benutzer" bezüglich Listenfenster: Dient der Auflistung der Benutzer. Dies ist die Beschreibung der Funktionalität des Moduls "user" mit dem Namen "Benutzer" bezüglich Selektionsfenster: Dieses Fenster dient der Selektion eines Benutzers. Dies ist die Beschreibung der Funktionalität des Moduls "user" mit dem Namen "Benutzer" bezüglich Benutzereinstellungen: Über dieses Fenster kann der in ClassiX® angemeldete Benutzer (User) seine eigenen Einstellungen setzen. Dadurch unterscheidet sich diese Fenster zum Editierfenster, in dem der Administrator die Einstellungen der User setzen kann. Im Gegensatz zum Editierfenster können über dieses Fenster administrative Einstellungen, wie beispielsweise die Zugriffsrechte, nicht geändert werden. Dies ist die Beschreibung der Funktionalität des Moduls "user" mit dem Namen "Benutzer" bezüglich Passwort setzen: Über dieses Fenster kann man einem Benutzer direkt ein Passwort geben. Dieses Passwort kann als temporäres oder als engültiges Passwort erfasst werden. Bei Vergabe eines temporären Passwortes muss der Benutzer nach Login dieses sofort in ein endgültiges Passwort ändern. Feld Beschreibung Benutzer : Neues Passwort Eingabe des neuen Passwortes. : Wiederholung Erneute Eingabe des neuen Passwortes, um beide Eingaben auf Gleichheit zu überprüfen. : als temporäres Passwort setzen Ist z.B. dann sinnvoll, wenn ein Benutzer nicht für sich selbst per E-Mail ein Passwort beantragen kann. Dann kann der Administrator dieses für den Benutzer tun, der Benutzer muss nach dem ersten Login das Passwort allerdings sofort ändern. Dies ist die Beschreibung der Funktionalität des Moduls "user" mit dem Namen "Benutzer" bezüglich Single Sign-on: Single Sign-on (SSO) definiert eine Funktionalität, die den Umgang mit der Vielzahl der Logins der genutzten Windows-Anwendungen durch Reduzierung auf nur eine einmalige Anmelderoutine erleichtern soll. Da zum Startzeitpunkt von ClassiX® bereits immer eine Windows-Anmeldung erfolgte, bezieht sich die ClassiX®-Single-Sign-on-Funktionalität auf diese. Um SSO in ClassiX® zu aktivieren, muß in der Benutzerverwaltung in dem Feld Windows SID der entsprechende Windows Security Identifier (SID) eingetragen werden. Wenn beim Start von ClassiX® genau 1 Übereinstimmung der dort hinterlegten Windows SID mit der Windows SID des Windows-Users übereinstimmt erfolgt das Login automatisch mit des Werten des gefundenen Users. Die SSO-Funktionalität greift bei den Varianten User + Login sowie Mandant + User + Login. Bei einem "Neu Anmelden" an ClassiX® wird die SSO-Funktionalität umgangen.
